# Рекомендуемые MQTT QoS и Retain Policy

| Сообщение | QoS | Retain | Почему |
|---|---:|---:|---|
| `open_cell` команда | `QoS 1` | `false` | Команда должна гарантированно дойти до ESP32, но не должна повторно выполниться после переподключения устройства. |
| `status` по команде | `QoS 1` | `false` | Сервер должен получить подтверждение выполнения команды и все переходы состояний по `request_id`. |
| `event normal_open` | `QoS 1` | `false` | Событие открытия ячейки должно быть доставлено без потери. |
| `event forced_open` | `QoS 1` | `false` | Критически важное событие безопасности, которое необходимо доставить без потери. |
| `error door_left_open` | `QoS 1` | `false` | Ошибка сценария закрытия двери должна быть доставлена без потерь. |
| `error relay_failure` | `QoS 1` | `false` | Ошибка исполнительного контура не должна теряться. |
| `heartbeat` | `QoS 0` | `false` | Периодическое сообщение о доступности устройства. Потеря отдельных heartbeat не является критичной. |
| `sensors status` | `QoS 0` | `true` или `false` | Может использовать `retain=true`, если необходимо сразу получать последние показания датчиков после подписки. |
| `device online/offline` | `QoS 1` | `true` | Позволяет новым подписчикам сразу узнать последнее известное состояние устройства. |

## Примечания

- Для всех управляющих команд рекомендуется использовать `QoS 1`.
- Команды управления замками (`open_cell`) **никогда не должны использовать `retain=true`**, чтобы избежать повторного открытия ячеек после переподключения ESP32.
- Для всех сообщений с жизненным циклом команды рекомендуется сохранять одинаковый `request_id` и не использовать `retain=true`.
- `QoS 2` в проекте GetBox не рекомендуется использовать, поскольку он увеличивает накладные расходы и не дает существенных преимуществ для данного сценария.
- `retain=true` следует применять только к информационным сообщениям, отражающим текущее состояние системы, но не к действиям, вызывающим физическое воздействие на оборудование.
